(TDD) ou em português <b>Desenvolvimento guiado por testes</b> é uma técnica de <a href=\"/wiki/Desenvolvimento_de_software\" title=\"Desenvolvimento de software\">desenvolvimento de software</a> que se relaciona com o conceito de <a href=\"/wiki/Verifica%C3%A7%C3%A3o_e_valida%C3%A7%C3%A3o\" title=\"Verificação e validação\">verificação e validação</a> e se baseia em um ciclo curto de repetições: Primeiramente o desenvolvedor escreve um <a href=\"/wiki/Caso_de_teste\" title=\"Caso de teste\">caso de teste</a> automatizado que define uma melhoria desejada ou uma nova funcionalidade. Então, é produzido código que possa ser validado pelo teste para posteriormente o código ser <a href=\"/wiki/Refatora%C3%A7%C3%A3o\" title=\"Refatoração\">refatorado</a> para um código sob padrões aceitáveis. <a href=\"/wiki/Kent_Beck\" title=\"Kent Beck\">Kent Beck</a>, considerado o criador ou o 'descobridor' da técnica, declarou em 2003 que TDD encoraja designs de código simples e inspira confiança.<sup id=\"cite_ref-Beck_1-0\" class=\"reference\"><a href=\"#cite_note-Beck-1\"><span>[</span>1<span>]</span></a></sup>\nDesenvolvimento dirigido por testes é relacionado a conceitos de programação de <a href=\"/wiki/Programa%C3%A7%C3%A3o_extrema\" title=\"Programação extrema\">Extreme Programming</a>, iniciado em 1999,<sup id=\"cite_ref-Cworld92_2-0\" class=\"reference\"><a href=\"#cite_note-Cworld92-2\"><span>[</span>2<span>]</span></a></sup> mas recentemente tem-se criado maior interesse pela mesma em função de seus próprios ideais.<sup id=\"cite_ref-Newkirk_3-0\" class=\"reference\"><a href=\"#cite_note-Newkirk-3\"><span>[</span>3<span>]</span></a></sup>\nAtravés de TDD, programadores podem aplicar o conceito de melhorar e <a externas</span></a></li>\n</ul>\n</div>\n\n<h2><span class=\"mw-headline\" id=\"Requisitos\">Requisitos</span><span class=\"mw-editsection\"><span class=\"mw-editsection-bracket\">[</span><a href=\"/w/index.php?title=Test-driven_development&amp;veaction=edit&amp;section=1\" class=\"mw-editsection-visualeditor\" title=\"Editar secção: Requisitos\">editar</a><span class=\"mw-editsection-divider\"> | </span><a href=\"/w/index.php?title=Test-driven_development&amp;action=edit&amp;section=1\" title=\"Editar secção: Requisitos\">editar código-fonte</a><span class=\"mw-editsection-bracket\">]</span></span></h2>\n<p>Desenvolvimento dirigido por testes<sup id=\"cite_ref-5\" class=\"reference\"><a href=\"#cite_note-5\"><span>[</span>5<span>]</span></a></sup> requer dos desenvolvedores criar <a href=\"/wiki/Teste_de_software\" title=\"Teste de software\">testes automatizados</a> que definam requisitos em código antes de escrever o código da aplicação. Os testes contém <a href=\"/w/index.php?title=Asser%C3%A7%C3%B5es_(computa%C3%A7%C3%A3o)&amp;action=edit&amp;redlink=1\" class=\"new\" title=\"Asserções (computação) (página não existe)\">asserções</a> que podem ser verdadeiras ou falsas. Após as mesmas serem consideradas verdadeiras após sua execução, os testes confirmam o comportamento correto, permitindo os desenvolvedores evoluir e <a href=\"/wiki/Refatora%C3%A7%C3%A3o\" title=\"Refatoração\">refatorar</a> o código. Normalmente todos os testes são efetuados de forma continua de acordo com o desenvolvimento cada funcionalidade criada deve ser acompanhada de um teste bem descrito e projetado, então deve-se escolher a área do projeto ou requisitos da tarefa para melhor orientar o desenvolvimento destes testes.\n</p><p>Desenvolvedores normalmente usam <a href=\"/w/index.php?title=Framework_de_testes&amp;action=edit&amp;redlink=1\" class=\"new\" title=\"Framework de testes (página não existe)\">frameworks de testes</a>, como <a href=\"/wiki/XUnit\" title=\"XUnit\">xUnit</a>, para criar e executar automaticamente uma série de casos de teste.\n</p>\n<h2><span id=\"Evolu.C3.A7.C3.A3o_hist.C3.B3rica\"></span><span class=\"mw-headline\" id=\"Evolução_histórica\">Evolução histórica</span><span class=\"mw-editsection\"><span class=\"mw-editsection-bracket\">[</span><a href=\"/w/index.php?title=Test-driven_development&amp;veaction=edit&amp;section=2\" class=\"mw-editsection-visualeditor\" title=\"Editar secção: Evolução histórica\">editar</a><span class=\"mw-editsection-divider\"> | </span><a href=\"/w/index.php?title=Test-driven_development&amp;action=edit&amp;section=2\" title=\"Editar secção: Evolução histórica\">editar código-fonte</a><span class=\"mw-editsection-bracket\">]</span></span></h2>\n<p>Kent Beck, considerado o criador ou o <i>descobridor</i> da técnica, declarou em 2003 que TDD encoraja designs de código simples e inspira confiança. Desenvolvimento dirigido por testes é relacionado a conceitos de programação de <i>Extreme Programming</i>, iniciado em <i>O desenvolvimento guiado por testes</i>, em língua inglesa <i>Test Driven Development</i> (TDD), teve origem na metodologia ágil criado por Kent Beck. Ela se tornou mais conhecida entre desenvolvedores após a publicação do livro TDD by exemplo de Kent Beck em 2002. Ainda que utilização já ocorresse há algumas décadas, segundo Larman e Basili (2003), o manifesto ágil contribuiu muito para impulsioná-la. próprios ideais. Através de TDD, programadores podem aplicar o conceito de melhorar e depurar código legado desenvolvido a partir de técnicas antigas.\n</p>\n<h2><span class=\"mw-headline\" id=\"Ciclo_de_desenvolvimento\">Ciclo de desenvolvimento</span><span class=\"mw-editsection\"><span class=\"mw-editsection-bracket\">[</span><a href=\"/w/index.php?title=Test-driven_development&amp;veaction=edit&amp;section=3\" class=\"mw-editsection-visualeditor\" title=\"Editar secção: Ciclo de desenvolvimento\">editar</a><span class=\"mw-editsection-divider\"> | </span><a href=\"/w/index.php?title=Test-driven_development&amp;action=edit&amp;section=3\" title=\"Editar secção: Ciclo de desenvolvimento\">editar código-fonte</a><span class=\"mw-editsection-bracket\">]</span></span></h2>\n<h3><span class=\"mw-headline\" id=\"1._Adicione_um_teste\">1. Adicione um teste</span><span class=\"mw-editsection\"><span class=\"mw-editsection-bracket\">[</span><a href=\"/w/index.php?title=Test-driven_development&amp;veaction=edit&amp;section=4\" class=\"mw-editsection-visualeditor\" title=\"Editar secção: 1. Adicione um teste\">editar</a><span class=\"mw-editsection-divider\"> | </span><a href=\"/w/index.php?title=Test-driven_development&amp;action=edit&amp;section=4\" title=\"Editar secção: 1. Adicione um teste\">editar código-fonte</a><span class=\"mw-editsection-bracket\">]</span></span></h3>\n<p>Para escrever um teste, o desenvolvedor precisa claramente entender as especificações e requisitos da funcionalidade. O desenvolvedor pode fazer isso através de caso de uso que cubram os requisitos e exceções condicionais. \n</p><p>Esta é a diferenciação entre desenvolvimento dirigido a testes entre escrever testes de unidade depois do código desenvolvido. Ele torna o desenvolvedor focado nos requisitos antes do código, que é uma sutil porem importante diferença.\n</p><p>Em um estudo por George e Willians (2003), com programadores constatou-se que: (87,5%) acreditavam que TDD facilita uma melhor abordagem e compreensão dos requisitos; 95,8% acreditavam que o tempo de depuração foi reduzido; 78% acreditavam que a produtividade do time aumentou; 92% acreditavam que melhorou a qualidade do código; 79% acreditavam que o design ficou mais simples; e no total 71% acreditavam que a abordagem foi eficaz.\n</p>\n<h3><span class=\"mw-headline\" id=\"2._Execute_todos_os_testes_e_veja_se_algum_deles_falha\">2. Execute todos os testes e veja se algum deles falha</span><span class=\"mw-editsection\"><span class=\"mw-editsection-bracket\">[</span><a href=\"/w/index.php?title=Test-driven_development&amp;veaction=edit&amp;section=5\" class=\"mw-editsection-visualeditor\" title=\"Editar secção: 2. Execute todos os testes e veja se algum deles falha\">editar</a><span class=\"mw-editsection-divider\"> | </span><a href=\"/w/index.php?title=Test-driven_development&amp;action=edit&amp;section=5\" title=\"Editar secção: 2. Execute todos os testes e veja se algum deles falha\">editar código-fonte</a><span class=\"mw-editsection-bracket\">]</span></span></h3>\n<p>Esse passo valida se todos os testes estão funcionando corretamente e se o novo teste não traz nenhum equívoco, sem requerer nenhum código novo. Pode-se considerar que este passo então testa o próprio teste: ele regula a possibilidade de novo teste passar.\n</p><p>O novo teste deve então falhar pela razão esperada: a funcionalidade não foi desenvolvida. Isto aumenta a confiança (por outro lado não exatamente a garante) que se está testando a coisa certa, e que o teste somente irá passar nos casos intencionados.\n</p>\n<h3><span id=\"3._Escrever_c.C3.B3digo\"></span><span class=\"mw-headline\" id=\"3._Escrever_código\">3. Escrever código</span><span class=\"mw-editsection\"><span class=\"mw-editsection-bracket\">[</span><a href=\"/w/index.php?title=Test-driven_development&amp;veaction=edit&amp;section=6\" class=\"mw-editsection-visualeditor\" title=\"Editar secção: 3. Escrever código\">editar</a><span class=\"mw-editsection-divider\"> | </span><a href=\"/w/index.php?title=Test-driven_development&amp;action=edit&amp;section=6\" title=\"Editar secção: 3. Escrever código\">editar código-fonte</a><span class=\"mw-editsection-bracket\">]</span></span></h3>\n<p>O próximo passo é escrever código que irá ocasionar ao teste passar. O novo código escrito até esse ponto poderá não ser perfeito e pode, por exemplo, passar no teste de uma forma não elegante. Isso é aceitável porque posteriormente ele será melhorado.\n</p><p>O importante é que o código escrito deve ser construído <i>somente</i> para passar no teste; nenhuma funcionalidade (muito menos não testada) deve ser predita ou permitida em qualquer ponto.\n</p>\n<h3><span class=\"mw-headline\" id=\"4._Execute_os_testes_automatizados_e_veja-os_executarem_com_sucesso\">4. Execute os testes automatizados e veja-os executarem com sucesso</span><span class=\"mw-editsection\"><span class=\"mw-editsection-bracket\">[</span><a href=\"/w/index.php?title=Test-driven_development&amp;veaction=edit&amp;section=7\" class=\"mw-editsection-visualeditor\" title=\"Editar secção: 4. Execute os testes automatizados e veja-os executarem com sucesso\">editar</a><span class=\"mw-editsection-divider\"> | </span><a href=\"/w/index.php?title=Test-driven_development&amp;action=edit&amp;section=7\" title=\"Editar secção: 4. Execute os testes automatizados e veja-os executarem com sucesso\">editar código-fonte</a><span class=\"mw-editsection-bracket\">]</span></span></h3>\n<p>Se todos os testes passam agora, o programador pode ficar confiante de que o código possui todos os requisitos testados. Este é um bom ponto que inicia o passo final do ciclo TDD.\n</p>\n<h3><span id=\"5._Refatorar_c.C3.B3digo\"></span><span class=\"mw-headline\" id=\"5._Refatorar_código\">5. Refatorar código</span><span class=\"mw-editsection\"><span class=\"mw-editsection-bracket\">[</span><a href=\"/w/index.php?title=Test-driven_development&amp;veaction=edit&amp;section=8\" class=\"mw-editsection-visualeditor\" title=\"Editar secção: 5. Refatorar código\">editar</a><span class=\"mw-editsection-divider\"> | </span><a href=\"/w/index.php?title=Test-driven_development&amp;action=edit&amp;section=8\" title=\"Editar secção: 5. Refatorar código\">editar código-fonte</a><span class=\"mw-editsection-bracket\">]</span></span></h3>\n<p>Nesse ponto o código pode ser limpo como necessário. Ao reexecutar os testes, o desenvolvedor pode confiar que a <a href=\"/wiki/Refatora%C3%A7%C3%A3o\" title=\"Refatoração\">refatoração</a> não é um processo danoso a qualquer funcionalidade existente. Um conceito relevante nesse momento é o de remoção de duplicação de código, considerado um importante aspecto ao design de um software. Nesse caso, entretanto, isso aplica remover qualquer duplicação entre código de teste e código de produção — por exemplo <a href=\"/w/index.php?title=Magic_number_(programming)&amp;action=edit&amp;redlink=1\" class=\"new\" title=\"Magic number (programming) (página não existe)\">magic numbers or strings</a> que nós repetimos nos testes e no código de produção, de forma que faça o teste passar no passo 3.\n</p>\n<h3><span class=\"mw-headline\" id=\"6._Repita_tudo\">6. Repita tudo</span><span class=\"mw-editsection\"><span class=\"mw-editsection-bracket\">[</span><a href=\"/w/index.php?title=Test-driven_development&amp;veaction=edit&amp;section=9\" class=\"mw-editsection-visualeditor\" title=\"Editar secção: 6. Repita tudo\">editar</a><span class=\"mw-editsection-divider\"> | </span><a href=\"/w/index.php?title=Test-driven_development&amp;action=edit&amp;section=9\" title=\"Editar secção: 6. Repita tudo\">editar código-fonte</a><span class=\"mw-editsection-bracket\">]</span></span></h3>\n<p>Iniciando com outro teste, o ciclo é então repetido, empurrando a funcionalidade a frente. O tamanho dos passos deve ser pequeno - tão quanto de 1 a 10 edições de texto entre cada execução de testes. Se novo código não satisfaz rapidamente um novo teste, ou outros testes falham inesperadamente, o programador deve <a href=\"/wiki/Desfazer\" title=\"Desfazer\">desfazer</a> ou reverter as alterações ao invés do uso de excessiva <a href=\"/wiki/Depura%C3%A7%C3%A3o\" title=\"Depuração\">depuração</a>. A <a href=\"/wiki/Integra%C3%A7%C3%A3o_cont%C3%ADnua\" title=\"Integração contínua\">integração contínua</a> ajuda a prover pontos reversíveis. É importante lembrar que ao usar bibliotecas externas não é interessante gerar incrementos tão pequenos que possam efetivamente testar a biblioteca ,<sup id=\"cite_ref-Newkirk_3-1\" class=\"reference\"><a href=\"#cite_note-Newkirk-3\"><span>[</span>3<span>]</span></a></sup> a menos que haja alguma razão para acreditar que a biblioteca tenha defeitos ou não seja suficientemente completada com suas funcionalidades de forma a servir às necessidades do programa em que está sendo escrito.\n</p>\n<h2><span class=\"mw-headline\" id=\"Estilos_de_desenvolvimento\">Estilos de desenvolvimento</span><span class=\"mw-editsection\"><span class=\"mw-editsection-bracket\">[</span><a href=\"/w/index.php?title=Test-driven_development&amp;veaction=edit&amp;section=10\" class=\"mw-editsection-visualeditor\" title=\"Editar secção: Estilos de desenvolvimento\">editar</a><span class=\"mw-editsection-divider\"> | </span><a href=\"/w/index.php?title=Test-driven_development&amp;action=edit&amp;section=10\" title=\"Editar secção: Estilos de desenvolvimento\">editar código-fonte</a><span class=\"mw-editsection-bracket\">]</span></span></h2>\n<p>Há vários aspectos ao usar desenvolvimento dirigido a testes, por exemplo os princípios \"Keep It Simple, Stupid\" (<a href=\"/wiki/KISS_principle\" class=\"mw-redirect\" title=\"KISS principle\">KISS</a>) e \"You Ain`t Gonna Need It\" (<a href=\"/wiki/YAGNI\" title=\"YAGNI\">YAGNI</a>). Focando em escrever código somente para os testes passarem, o design do sistema pode ser mais limpo e claro do que o que é alcançado por outros métodos.<sup id=\"cite_ref-Beck_1-1\" class=\"reference\"><a href=\"#cite_note-Beck-1\"><span>[</span>1<span>]</span></a></sup> Em <i>Test-Driven Developmente By Example</i> Kent Beck sugere o princípio \"Fake it, till you make it\".\nPara alcançar altos níveis conceituais de design (como o uso de <a href=\"/w/index.php?title=Design_Pattern&amp;action=edit&amp;redlink=1\" class=\"new\" title=\"Design Pattern (página não existe)\">Design Pattern</a>), testes são escritos de forma que possam gerar o design. O código pode acabar permanecendo mais simples do que o padrão alvo, entretanto ele deve passar em todos os testes requeridos. Isto pode ser inaceitável de primeira mas ele permite o desenvolvedor focar somente no que é importante.\n</p><p><b>Falhar primeiro os casos de testes</b>. A ideia é garantir que o teste realmente funciona e consegue capturar um erro. Assim que ele é mostrado, a funcionalidade almejada pode ser implementada. Isto tem sido apontado como o \"Test-Driven Mantra\", conhecido como vermelho/verde/refatorar onde vermelho significa <i>falhar</i> onde pelo menos uma asserção falha ,e verde é <i>passar</i>, que significa que todas as asserções foram verdadeiras.\n</p><p>Desenvolvimento dirigido a testes constantemente repete os passos de adicionar casos de teste que falham, passando e refatorando-os. Ao receber o resultado esperado em cada estágio reforça ao desenvolvedor o modelo mental do código, aumentando a confiança e incrementando a produtividade.\n</p><p>Práticas avançadas de desenvolvimento dirigido a testes encaminham para o desenvolvimento dirigido a testes de aceitação (ATDD), onde o critério especificado pelo cliente é automatizado em testes de aceitação, que então levam ao tradicional processo de desenvolvimento dirigido a testes de unidade. Este processo garante que o cliente tem um mecanismo automatizado para decidir como o software atende suas necessidades. Com ATDD, o desenvolvimento em equipe tem objetivo específico para satisfazer, e os testes de aceitação os mantém continuamente focados em o que o cliente realmente deseja daquela funcionalidade.\n</p>\n<h2><span id=\"Benef.C3.ADcios\"></span><span class=\"mw-headline\" id=\"Benefícios\">Benefícios</span><span class=\"mw-editsection\"><span class=\"mw-editsection-bracket\">[</span><a href=\"/w/index.php?title=Test-driven_development&amp;veaction=edit&amp;section=11\" class=\"mw-editsection-visualeditor\" title=\"Editar secção: Benefícios\">editar</a><span class=\"mw-editsection-divider\"> | </span><a href=\"/w/index.php?title=Test-driven_development&amp;action=edit&amp;section=11\" title=\"Editar secção: Benefícios\">editar código-fonte</a><span class=\"mw-editsection-bracket\">]</span></span></h2>\n<p>Um estudo de 2005 descobriu que usar TDD significava escrever mais testes, e logo, programadores que escreviam mais testes tendiam a ser mais produtivos.<sup id=\"cite_ref-6\" class=\"reference\"><a href=\"#cite_note-6\"><span>[</span>6<span>]</span></a></sup> Hipóteses relacionando a qualidade de código e uma correlação direta entre TDD e produtividade foram inconclusivas.<sup id=\"cite_ref-7\" class=\"reference\"><a href=\"#cite_note-7\"><span>[</span>7<span>]</span></a></sup>\n</p><p>Desenvolvedores usando TDD puramente em novos projetos reportaram que raramente necessitaram a utilização de um <a href=\"/wiki/Depurador\" title=\"Depurador\">depurador</a>. Usado em junção com um <a href=\"/wiki/Sistema_de_controle_de_vers%C3%A3o\" class=\"mw-redirect\" title=\"Sistema de controle de versão\">Sistema de controle de versão</a>, quando testes falham inesperadamente, reverter o código para a última versão em que os testes passaram pode ser mais produtivo do que depurar.<sup id=\"cite_ref-8\" class=\"reference\"><a href=\"#cite_note-8\"><span>[</span>8<span>]</span></a></sup><sup id=\"cite_ref-9\" class=\"reference\"><a href=\"#cite_note-9\"><span>[</span>9<span>]</span></a></sup>\n</p><p>Desenvolvimento dirigido por testes oferece mais do que somente um maneira simples de validação e de correção, pode orientar o design de um programa. Por focar em casos de testes primeiramente, deve-se imaginar como a funcionalidade será usada pelo cliente. Logo, o programador é somente com a interface e não com a implementação. Este benefício é complementar ao <a href=\"/w/index.php?title=Design_por_Contrato&amp;action=edit&amp;redlink=1\" class=\"new\" title=\"Design por Contrato (página não existe)\">Design por Contrato</a>, que através torna os casos de testes muito mais do que asserções matemáticas ou pré-concepções.\n</p><p>O poder que TDD oferece é a habilidade de pegar pequenas partes quando requeridas. Isso permite o desenvolvedor focar como objetivo fazer os testes atuais passarem. Casos excepcionais e tratamento de erros não são considerados inicialmente. Testes que criam estas circunstâncias estranhas são implementadas separadamente. Outra vantagem é que TDD quando usado apropriadamente, garante que todo o código desenvolvido seja coberto por um teste. Isto fornece a equipe de desenvolvimento, e ao usuários, subsequentemente, um grande nível de confiança ao código.\n</p><p>Enquanto que é verdade que mais código é necessário ao usar TDD do que sem TDD, devido aos códigos de teste, o tempo total de implementação é tipicamente menor.<sup id=\"cite_ref-10\" class=\"reference\"><a href=\"#cite_note-10\"><span>[</span>10<span>]</span></a></sup> Um grande número de testes ajudam a limitar o número de defeitos no código. A natureza periódica ajuda a capturar defeitos inicialmente no ciclo de desenvolvimento, prevenindo-os de se tornarem grandes e endêmicos problemas. Eliminando defeitos cedo no processo normalmente evita longos e tediantes períodos de depuração posteriores em um projeto.\n</p><p>TDD pode encaminhar a um nível que possibilite um código mais modularizado, flexível e extensível. Este efeito surge devido a metodologia requerer que os desenvolvedores pensem no software em pequenas unidades que podem ser reescritas, desenvolvidas e testadas independentemente e integradas depois. Isto implica menores e mais classes, evitando o acoplamento e permitindo interfaces mais limpas. O uso de <a href=\"/wiki/Objeto_mock\" title=\"Objeto mock\">Mock Object</a> é um contribuidor para a modularização do código, pois este recurso requer que o código seja escrito de forma que possa ser facilmente trocado entre versões Mock, usados em testes de unidade, e \"reais\", usados na aplicação.\n</p><p>Devido a fato de que nenhum código é escrito a não ser para passar em um teste que esteja falhando, testes automatizados tendem a cobrir cada caminho de código. Por exemplo, para que um desenvolvedor possa adicionar um caminho alternativo \"senão\" em um \"se\" , o desenvolvedor poderia primeiramente escrever um teste que motive o fluxo alternativo. Como resultado, os testes automatizados TDD tendem a ser mais perfeitos: eles irão mostrar qualquer mudança inesperada no comportamento do código. Isto ajuda a identificar problemas cedo que poderiam aparecer ao consertar uma funcionalidade que ao modificada, inesperadamente altera outra funcionalidade.\n</p>\n<h2><span id=\"Limita.C3.A7.C3.B5es\"></span><span class=\"mw-headline\" id=\"Limitações\">Limitações</span><span class=\"mw-editsection\"><span class=\"mw-editsection-bracket\">[</span><a href=\"/w/index.php?title=Test-driven_development&amp;veaction=edit&amp;section=12\" class=\"mw-editsection-visualeditor\" title=\"Editar secção: Limitações\">editar</a><span class=\"mw-editsection-divider\"> | </span><a href=\"/w/index.php?title=Test-driven_development&amp;action=edit&amp;section=12\" title=\"Editar secção: Limitações\">editar código-fonte</a><span class=\"mw-editsection-bracket\">]</span></span></h2>\n<ol><li>Desenvolvimento dirigido com testes é difícil de usar em situações onde testes totalmente funcionais são requeridos para determinar o sucesso ou falha. Exemplos disso são interfaces de usuários, programas que trabalham com base de dados, e muitos  outros que dependem de configurações específicas de rede. TDD encoraja os desenvolvedores a incluir o mínimo de código funcional em módulos e maximizar a lógica, que é extraída em código de teste, usando <a href=\"/wiki/Fakes\" class=\"mw-redirect\" title=\"Fakes\">Fakes</a> mocks para representar o mundo externo.</li>\n<li>Suporte gerencial é essencial. Se toda a organização não acreditar que TDD é para melhorar o produto, o gerenciamento irá considerar que o tempo gasto escrevendo teste é desperdício.<sup id=\"cite_ref-11\" class=\"reference\"><a href=\"#cite_note-11\"><span>[</span>11<span>]</span></a></sup></li>\n<li>Os próprios testes se tornam parte da manutenção do projeto. Testes mal escritos, por exemplo, que incluem strings de erro embutidas ou aqueles que são suceptíveis a falha, são caros de manter. Há um risco em testes que geram falsas falhas de tenderem a serem ignorados. Assim quando uma falha real ocorre, ela pode não ser detectada. É possível escrever testes de baixa e fácil manutenção, por exemplo pelo reuso das strings de erro, podendo ser o objetivo durante a fase de refatoração descrita acima.</li>\n<li>O nível de cobertura e detalhamento de teste alcançado durante repetitivos ciclos de TDD não pode ser facilmente recriado em uma data tardia. Com o passar do tempo os testes vão se tornando gradativamente preciosos. Se uma arquitetura pobre, um mal design ou uma estratégia de teste mal feita acarretar em mudança tardia, fazendo com que dezenas de testes falhem, por outro lado eles são individualmente consertáveis. Entretanto, simplesmente deletando, desabilitando ou alterando-os vagamente poderá criar buracos indetectáveis na cobertura de testes.</li>\n<li>Lacunas inesperadas na cobertura de teste podem existir ou ocorrer por uma série de razões. Talvez um ou mais desenvolvedores em uma equipe não foram submetidos ao uso de TDD e não escrevem testes apropriadamente, talvez muitos conjuntos de testes foram invalidados, excluídos ou desabilitados acidentalmente ou com o intuito de melhorá-los posteriormente. Se isso acontece, a certeza é de que um enorme conjunto de testes TDD serão corrigidos tardiamente e refatorações serão mal acopladas. Alterações podem ser feitas não resultando em falhas, entretanto, na verdade, bugs estão sendo introduzidos imperceptivelmente, permanecendo indetectáveis.</li>\n<li>Testes de unidade criados em um ambiente de desenvolvimento dirigido por testes são tipicamente criados pelo desenvolvedor que irá então escrever o código que está sendo testado. Os testes podem consequentemente compartilhar os 'pontos cegos' no código: Se por exemplo, um desenvolvedor não realizar determinadas entradas de parâmetros que precisam ser checadas, muito provavelmente nem o teste nem o código irá verificar essas entradas. Logo, se o desenvolvedor interpreta mal a especificação dos requisitos para o módulo que está sendo desenvolvido, tanto os testes como o código estarão errados.</li>\n<li>O alto número de testes de unidades pode trazer um falso senso de segurança, resultando em menor nível de atividades de garantia de qualidade, como testes de integração e aceitação.</li></ol>\n<h2><span id=\"Visibilidade_de_c.C3.B3digo\"></span><span class=\"mw-headline\" id=\"Visibilidade_de_código\">Visibilidade de código</span><span class=\"mw-editsection\"><span class=\"mw-editsection-bracket\">[</span><a href=\"/w/index.php?title=Test-driven_development&amp;veaction=edit&amp;section=13\" class=\"mw-editsection-visualeditor\" title=\"Editar secção: Visibilidade de código\">editar</a><span class=\"mw-editsection-divider\"> | </span><a href=\"/w/index.php?title=Test-driven_development&amp;action=edit&amp;section=13\" title=\"Editar secção: Visibilidade de código\">editar código-fonte</a><span class=\"mw-editsection-bracket\">]</span></span></h2>\n<p>O conjunto de teste claramente possibilita acessar o código que está se testando. Por outro lado critérios normais de design como <a href=\"/w/index.php?title=Information_hiding&amp;action=edit&amp;redlink=1\" class=\"new\" title=\"Information hiding (página não existe)\">Information hiding</a>, encapsulamento e <a href=\"/wiki/Separa%C3%A7%C3%A3o_de_conceitos\" title=\"Separação de conceitos\">separação de conceitos</a> não devem ser comprometidos. Consequentemente teste de unidade são normalmente escritos no mesmo projeto ou  <a href=\"/wiki/M%C3%B3dulo_(programa%C3%A7%C3%A3o)\" class=\"mw-redirect\" title=\"Módulo (programação)\">módulo</a> que o código está sendo testado.\n</p><p>Ao usar <a href=\"/w/index.php?title=Design_orientado_a_objetos&amp;action=edit&amp;redlink=1\" class=\"new\" title=\"Design orientado a objetos (página não existe)\">design orientado a objetos</a>, este não provê acesso ao métodos e dados privados. Consequentemente, trabalho extra pode ser necessário para testes de unidade. Em <a href=\"/wiki/Java_(linguagem_de_programa%C3%A7%C3%A3o)\" title=\"Java (linguagem de programação)\">Java</a> e outras liguagens, um desenvolvedor pode usar <a href=\"/w/index.php?title=Reflex%C3%A3o_(ci%C3%AAncia_da_computa%C3%A7%C3%A3o)&amp;action=edit&amp;redlink=1\" class=\"new\" title=\"Reflexão (ciência da computação) (página não existe)\">reflexão</a> para acessar campos que são marcados como privados.<sup id=\"cite_ref-12\" class=\"reference\"><a href=\"#cite_note-12\"><span>[</span>12<span>]</span></a></sup>\nAlternativamente, uma classe inerente pode ser usada para suportar os testes de unidade, assim tendo visibilidade dos membros e atributos da classe. No <a href=\"/wiki/.NET_Framework\" title=\".NET Framework\">.NET</a> e em muitas outras linguagens, <a href=\"/w/index.php?title=Classes_parciais&amp;action=edit&amp;redlink=1\" class=\"new\" title=\"Classes parciais (página não existe)\">classes parciais</a> podem ser usadas para expor métodos privados e dados para os testes acessarem.\n</p><p>O importante é que modificações brutas no testes não apareçam no código de produção. Em <a href=\"/w/index.php?title=C_Sharp_(linguage_de_programa%C3%A7%C3%A3o)&amp;action=edit&amp;redlink=1\" class=\"new\" title=\"C Sharp (linguage de programação) (página não existe)\">C#</a> e em outras linguagens, diretivas de de pré-compilação como <code>#if DEBUG ... #endif</code> pode ser posicionadas em volta de classes adicionais e realmente prevenir todos os outros códigos relacionados a teste de serem compilados no código de lançamento. Isto então significa que o código lançado não é exatamente o mesmo de quando testado. A execução regular de poucos mais completos,  testes de integração de aceitação ao final do lançamento do código podem garantir que não existe código de produção que subitamente dependa dos testes.\nHá muito debate entre os praticantes de TDD, documentados nos seus blogs e outros locais, com a dúvida se é de bom julgamento testar métodos privados e protegidos e dados de qualquer maneira. Muitos argumentam que poderia ser suficiente testar qualquer classe através da interface pública, considerando membros privados como meramente detalhes de implementações que podem mudar, e se deveria ser permitido fazer sem quebrar nenhum teste. Outros dizem que aspectos cruciais de uma funcionalidade podem ser implementados em métodos privados, e que ao fazer desta forma eles são indiretamente testados através da interface pública obscurecendo a ocorrência dos mesmos: Testes de unidade são para testar a menor unidade de funcionalidade possível.<sup id=\"cite_ref-13\" class=\"reference\"><a href=\"#cite_note-13\"><span>[</span>13<span>]</span></a></sup><sup id=\"cite_ref-14\" class=\"reference\"><a href=\"#cite_note-14\"><span>[</span>14<span>]</span></a></sup>\n</p>\n<h2><span id=\"Fakes.2C_mocks_e_testes_de_integra.C3.A7.C3.A3o\"></span><span class=\"mw-headline\" id=\"Fakes,_mocks_e_testes_de_integração\">Fakes, mocks e testes de integração</span><span class=\"mw-editsection\"><span class=\"mw-editsection-bracket\">[</span><a href=\"/w/index.php?title=Test-driven_development&amp;veaction=edit&amp;section=14\" class=\"mw-editsection-visualeditor\" title=\"Editar secção: Fakes, mocks e testes de integração\">editar</a><span class=\"mw-editsection-divider\"> | </span><a href=\"/w/index.php?title=Test-driven_development&amp;action=edit&amp;section=14\" title=\"Editar secção: Fakes, mocks e testes de integração\">editar código-fonte</a><span class=\"mw-editsection-bracket\">]</span></span></h2>\n<p>Testes de unidades são assim chamados por cada teste exercitar uma <i>unidade</i> de código. Se um módulo tem centenas de testes de unidade ou somente cinco é irrelevante. Um conjunto de testes em TDD nunca cruza os limites de um programa, nem deixa conexões de rede perdidas. Ao fazer essas ações, o mesmo introduz intervalos de tempo que podem tornar testes lentos ao serem executados, desencorajando desenvolvedores de executar toda a suite de testes. Introduzir dependências de módulos externos ou data transforma <i>teste de unidade</i> em <i>testes de integração</i>. Se um módulo se comporta mal em uma cadeia de módulos interrelacionados, não fica imediatamente claro onde olhar a causa da falha.\n</p><p>Quando o código em desenvolvimento depende confiavelmente de uma base de dados, um serviço web ou qualquer outro processo externo ou serviço, obrigando em um separação unitária de teste é então uma oportunidade forçada de criar um design de código mais modular, mais testável e reusável.<sup id=\"cite_ref-15\" class=\"reference\"><a href=\"#cite_note-15\"><span>[</span>15<span>]</span></a></sup> Dois passos são necessários:\n</p>\n<ol><li>Toda vez que um acesso externo é necessário no design final, uma <a href=\"/wiki/Interface_(ci%C3%AAncia_da_computa%C3%A7%C3%A3o)\" title=\"Interface (ciência da computação)\">interface</a> deve ser definida de forma a descrever que acessos irão ser disponíveis. O <a href=\"/w/index.php?title=Princ%C3%ADpio_de_invers%C3%A3o_de_depend%C3%AAncia&amp;action=edit&amp;redlink=1\" class=\"new\" title=\"Princípio de inversão de dependência (página não existe)\">princípio de inversão de dependência</a> fornece benefícios nessa situação em TDD.</li>\n<li>A interface deve ser implementada de duas maneiras, uma que realmente acessa o processo externo, e outra que é um <a href=\"/wiki/Objeto_Mock\" class=\"mw-redirect\" title=\"Objeto Mock\">fake ou mock</a>. Objetos fake precisam fazer um pouco mais do que adicionar mensagens \"Objeto Pessoa salvo\" criando <a href=\"/w/index.php?title=Registro_de_rastreio&amp;action=edit&amp;redlink=1\" class=\"new\" title=\"Registro de rastreio (página não existe)\">registro de rastreio</a>, identificando que asserções podem executadas para verificar o comportamento correto. Objetos mock diferem pelo fato que eles mesmos contém asserções que podem fazer com que o teste falhe. Exemplo:</li></ol>\n<dl><dd>Se o nome da pessoa e outro dado não é como esperado. métodos de objetos fake e mock que retornem dados, aparentemente de uma armazenamento de dados ou de usuário,pode ajudar ao processo de teste sempre retornar o mesmo, dados que testes pode confiar. Eles podem ser muito usáveis em predefinidos modos de falha em que rotinas de tratamento de erro pode ser desenvolvidas e confiavelmente testadas. Serviços fake dentre outros armazenamento de dados podem ser usáveis em TDD: um serviço fake de criptografia pode não, criptografar o dado passado;<sup class=\"noprint Inline-Template\" style=\"color:gray;white-space:nowrap;\">&#91;<i><a href=\"/wiki/Wikip%C3%A9dia:Por_favor,_seja_claro\" title=\"Wikipédia:Por favor, seja claro\"><span title=\"O texto exige esclarecimento ou retirada de jargões\">necessário esclarecer</span></a></i>&#93;</sup> serviços fake de número aleatório podem sempre retornar 1. Implementações de fakes e mocks são exemplos de <a href=\"/wiki/Inje%C3%A7%C3%A3o_de_depend%C3%AAncia\" title=\"Injeção de dependência\">injeção de dependência</a>.</dd></dl>\n<p>A proposta da injeção de dependência é que a base de dados ou qualquer ou código de acesso externo nunca é testado pelo processo TDD. Para desviar de erros que possam aparecer em função disso, outros testes com a real implementação das interfaces discutidas acima são necessários. Esses testes são separados dos testes unitários e são realmente <a href=\"/w/index.php?title=Testes_de_integra%C3%A7%C3%A3o&amp;action=edit&amp;redlink=1\" class=\"new\" title=\"Testes de integração (página não existe)\">testes de integração</a>. Eles serão poucos, e eles precisam ser menos executados os testes de unidades. Eles podem ser implementados sem nenhum problema usando o mesmo framework de testes, como <a href=\"/wiki/XUnit\" title=\"XUnit\">xUnit</a> .\n</p><p>Testes de integração que alteram qualquer <a href=\"/wiki/Armazenamento_persistente\" class=\"mw-redirect\" title=\"Armazenamento persistente\">armazenamento persistente</a> ou banco de dados deve ser designado cuidadosamente, levando em consideração os estados iniciais e finais dos arquivos e base de dados, independente se um teste falha. Isto é muitas vezes adquirido usando várias combinações das seguintes técnicas:\n</p>\n<ul><li>O método <code>TearDown</code>, que é integrado ao muitos frameworks de teste.</li>\n<li>Estruturas <code>try...catch...finally</code> de <a href=\"/wiki/Tratamento_de_exce%C3%A7%C3%B5es\" class=\"mw-redirect\" title=\"Tratamento de exceções\">tratamento de exceções</a>, quando disponíveis.</li>\n<li><a href=\"/w/index.php?title=Transa%C3%A7%C3%B5es_de_banco_de_dados&amp;action=edit&amp;redlink=1\" class=\"new\" title=\"Transações de banco de dados (página não existe)\">Transações de banco de dados</a> onde ela <a href=\"/wiki/Atomicidade\" class=\"mw-redirect\" title=\"Atomicidade\">atomicamente</a> inclui talvez um escrita, uma leitura e uma operação de deleção.</li>\n<li>Capturando o \"estado\" do banco de dados antes de executar qualquer teste e após a execução, desfasar para o \"estado\" antes do teste ser executado. Isso pode ser automatizado usando a framework como o <a href=\"/wiki/Apache_Ant\" title=\"Apache Ant\">Ant</a> ou <a href=\"/w/index.php?title=NAnt&amp;action=edit&amp;redlink=1\" class=\"new\" title=\"NAnt (página não existe)\">NAnt</a> ou um sistema de <a href=\"/wiki/Integra%C3%A7%C3%A3o_cont%C3%ADnua\" title=\"Integração contínua\">integração contínua</a> como o <a href=\"/w/index.php?title=CruiseControl&amp;action=edit&amp;redlink=1\" class=\"new\" title=\"CruiseControl (página não existe)\">CruiseControl</a>.</li>\n<li>Inicializando o banco de dados para um estado limpo <i>antes</i> dos testes, ao invés de limpar <i>depois</i> da execução dos testes. Isto pode ser relevante onde limpar depois pode tornar difícil de diagnosticar falhas de teste ao deletar o estado final do banco de dados antes que um diagnótico detalhado possa ser feito.</li></ul>\n<p>Framework como <a href=\"/w/index.php?title=Moq&amp;action=edit&amp;redlink=1\" class=\"new\" title=\"Moq (página não existe)\">Moq</a>, jMock, <a href=\"/w/index.php?title=NMock&amp;action=edit&amp;redlink=1\" class=\"new\" title=\"NMock (página não existe)\">NMock</a>, EasyMock, TypeMock, jMockit, Mockito, PowerMock e Rhino Mocks existem para tornar o processo de criar e usar objetos mock complexos facilmente.\n</p>\n<h2><span id=\"Ver_tamb.C3.A9m\"></span><span class=\"mw-headline\" id=\"Ver_também\">Ver também</span><span
